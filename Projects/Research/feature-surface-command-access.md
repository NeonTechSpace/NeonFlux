# Feature Surface And Command Access Notes

## Decisions

- Dashboard access is still Manage Server by default. Command grants do not grant dashboard access.
- Dashboard-only configuration is the default for feature setup, especially logging destinations, posting, autorole, reaction roles, verification, tickets, VC generator setup, import/export, and command access.
- Bot commands should exist only for chat-native user actions. Planned dashboard configuration must not create fake hidden bot commands.
- Command access grants target either a command category or a specific bot command:
    - `target_type = category`, example `target_id = settings`
    - `target_type = command`, example `target_id = settings.prefix`
- Old category grants are preserved as category targets during migration. Prefix command authorization also checks the legacy `prefix` category target so existing grants keep working.
- Grants apply only to real implemented guarded bot commands. Public help/ping stay discoverable, are DEFCON-gated, and are not grantable.
- Moderation V1 is live as guarded bot commands for case history, warnings, reasons, notes, kick, ban, unban, timeout, untimeout, and purge. Automod has dashboard-configured matching plus record/delete-message/timeout/warn actions, ignore lists, and privileged-author mutation skips.
- DEFCON 1/2 are stronger than grants. DEFCON 3 allows server owner, Manage Server, category grant, or command grant for guarded commands.
- Help output lists implemented visible commands regardless of whether the caller can execute them. Discovery does not bypass DEFCON.

## Feature Classification

- `general`: command prefix dashboard config plus `general.help`, `general.ping`, and `settings.prefix`.
- `autorole`: dashboard-only role rules plus a `member.joined` handler. No bot command.
- `reaction_roles`: dashboard-only message/emoji/role mapping plus `reaction.added` and `reaction.removed` handlers. No bot command.
- `verification`: dashboard-only message/emoji/verified-role flows plus `reaction.added` and `member.joined` handlers. No bot command.
- `role_reconciliation`: dashboard-configured role repair/cleanup policy plus `member.updated` repair and `role.deleted` cleanup handlers. No bot command.
- `logging`: dashboard-only destination/filter config plus server-event handlers. No bot command.
- `posting`: dashboard-only sender, embed builder, and saved message templates. No bot command.
- `profile_builder`: dashboard-configured profile forms plus a public authenticated member submission route and dashboard review surface. No bot command.
- `import_export`: dashboard-only structure export, import dry-run, guarded apply controls, and structure-change observation. No bot command.
- `invite_tracking`: dashboard-only invite analytics and member-flow tracking. No bot command.
- `giveaways`: dashboard-configured publish/close/reroll/cancel flow plus reaction entry tracking. No bot command.
- `suggestions`: dashboard-configured boards plus public `suggest <idea>` command and vote reactions. The command is DEFCON-gated, visible in help, and not grantable.
- `tickets`: dashboard-configured ticket panels plus reaction-based ticket opening and channel-deletion cleanup. No bot command.
- `vc_generator`: dashboard-only setup plus a bot-managed panel embed in a configured channel. Until interactions exist, panel controls use reactions for rename, user limit, whitelist, blacklist, lock, and unlock.
- `xp`: dashboard-configured message and VC activity rules, public `rank` and `leaderboard` commands, and event handlers for message/voice XP. Message activity and VC activity both count as XP sources.
- `moderation` now has implemented visible commands for `warn`, `kick`, `ban`, `unban`, `timeout`, `untimeout`, `purge`, `warnings`, `delwarn`, `clearwarn`, `case`, `cases`, `reason`, `note`, and `notes`.
- `automod`: dashboard-configured blocked-term and invite-link rules plus a hidden `message.created` handler that records matches. No bot command.
- Planned bot command groups remain hidden until implemented; tickets remains dashboard/panel-driven with no live chat command in the catalog.
- The moderation dashboard category is active for case history and moderation policy controls.
- Moderation policy is dashboard-configured through `guild_feature_settings` feature `moderation`. It protects configured user IDs and role IDs from `kick`, `ban`, and `unban`.

## Current Implementation Loop

- Landed: guarded `kick`, `ban`, `unban`, `timeout`, and `untimeout` bot commands create moderation cases, call the Fluxer moderation platform, record `action.applied` or `action.failed`, then resolve or void the case.
- Landed: the dashboard moderation route reads recent cases through the Manage Server dashboard authorization path and displays a read-only case table.
- Landed: moderation policy can be edited from the dashboard and writes a `bot_action_events` audit entry with `feature = moderation` and `action = policy.updated`.
- Landed: bot punishment commands check protected users first, then protected roles through the normalized Fluxer member-read port before creating a case or calling a destructive platform operation.
- Landed: `ban.added` and `ban.removed` events reconcile into moderation case history. Recent matching open/resolved cases are treated as command-initiated or already-observed work to avoid duplicate cases; unmatched events create resolved observed `ban` or `unban` cases.
- Landed: moderation case changes emit dashboard live invalidation events for the moderation page. These are not dashboard audit events and stay separate from the future Discord/server-event logging feature.
- Landed: guarded `purge` deletes up to 100 recent messages before the command message, records a channel-targeted moderation case, resolves it on Fluxer bulk-delete success, and voids it on platform failure. No live destructive server testing was performed.
- Landed: moderation cases now have an explicit target model: user-targeted cases keep `target_user_id`, while channel-targeted cases use `target_type = channel` plus `target_channel_id`. This keeps purge case history honest and avoids storing channel IDs in user fields.
- Landed: Fluxer timeout support is wrapped through member edit using `communication_disabled_until` and `timeout_reason`. Command durations accept `m`, `h`, and `d` units from 1 minute through 28 days.
- Landed: timeout creates a pending `moderation_temporary_actions` row for traceability, and new timeouts or manual untimeouts cancel stale pending timeout rows for the same user. Platform timeouts auto-expire at the Fluxer/Discord layer, so no live expiry mutation test was performed.
- Landed: logging destinations are dashboard-configured per event group in `guild_logging_destinations`. Config changes write dashboard audit events with `feature = logging` and emit logging live invalidation events.
- Landed: server-event logging handlers route message edits/deletes, member joins/updates/leaves, bans/unbans, role changes, channel changes, and voice-state updates to the configured destination channel. Log output uses Fluxer/Discord mentions where possible and keeps raw IDs for traceability.
- Landed: logging remains separate from dashboard audit events and does not create bot commands or help entries. If no destination is configured, loggable events are ignored; if a destination is configured and sending fails, the router returns `message-send-error`.
- Landed: autorole rules are dashboard-configured in the Access category. The dashboard reads bot-visible guild roles through `FLUXER_BOT_TOKEN`, saves enabled/disabled role rules, and records `bot_action_events` with `feature = autorole`.
- Landed: autorole applies configured roles on `member.joined` only after growth tracking records the join. It reads guild structure and the bot member's roles, then applies only roles below the bot's highest role; unsafe or already-present roles are skipped. No live role mutation testing was performed.
- Landed: reaction-role messages are dashboard-built in the Access category with channel selection, plain/embed payloads, generated option overviews, normal multi-claim mode, and exclusive one-from-menu mode. Config changes write `bot_action_events` with `feature = reaction_roles` and emit reaction-role live invalidation events.
- Landed: reaction add/remove events look up enabled non-stale reaction-role messages by guild, message, and emoji key. The bot applies or removes only roles below its highest role and records assignment state in `reaction_role_assignments`; exclusive mode removes the prior active assignment for that user/message before applying the new role. Option deletion removes the configured DB option and calls Fluxer's emoji-wide reaction cleanup for the already-sent message. No live role or reaction mutation testing was performed.
- Landed: verification flows are dashboard-configured in the Access category using a channel, message ID, emoji key, and verified role. Config changes write `bot_action_events` with `feature = verification` and emit verification live invalidation events.
- Landed: verification reaction events apply the configured verified role through the shared role-safety path, then record an active `verification_records` row. Verified members who rejoin have enabled verified roles restored if they are still safe for the bot to manage. No live role mutation testing was performed.
- Landed: role reconciliation repairs member role drift on `member.updated` by re-applying missing roles owned by enabled autorole rules, active verification records, and active reaction-role assignments. It records `role_reconciliation_runs` and `role_reconciliation_actions`, uses the shared bot role-hierarchy safety check, and never removes roles. No live role mutation testing was performed.
- Landed: `role.deleted` now runs a DB-only deleted-role reference cleanup before server-event logging. It disables matching autorole rules and verification flows, removes deleted roles from moderation protected roles, ticket staff roles, command/dashboard access grants, and XP role rewards, deletes matching reaction-role options, and marks active reaction-role assignments removed. It records a `role_reconciliation` run/action when anything changed and does not call Fluxer platform role/channel/message mutation APIs.
- Landed: role reconciliation settings are editable from the Access dashboard category. Settings default to the previous enabled behavior, can disable the full reconciliation feature, and can independently control autorole repair, verification repair, reaction-role repair, and deleted-role reference cleanup. Config changes write `bot_action_events` with `feature = role_reconciliation` and emit role-reconciliation live invalidation events.
- Landed: autorole and command/dashboard access DB rows now emit dashboard live invalidation events so Access pages can refresh when cleanup or another writer changes role-backed settings.
- Landed: command access is editable from the Access dashboard category. Manage Server dashboard users can grant bot command categories or specific guarded bot commands to user IDs and bot-visible roles, remove grants, and see role names when `FLUXER_BOT_TOKEN` lets the web service read server roles. This changes bot command authorization only; dashboard access still requires Manage Server.
- Landed: XP rules are dashboard-configured in the Community category. Settings control enablement, message XP range, message cooldown, voice XP per minute, and minimum voice-session duration. Config changes write `bot_action_events` with `feature = xp` and emit XP live invalidation events.
- Landed: XP has public visible `rank [user]` and `leaderboard` commands. They are DEFCON-gated public commands, visible in help, and not dashboard-grantable because they do not require Manage Server by default.
- Landed: message XP is awarded only for non-command messages when XP is enabled. Award amounts are deterministic per message ID within the configured range, cooldowns use the latest user XP projection, and grants are idempotent by message ID.
- Landed: voice XP uses persisted voice sessions. Joining or moving voice channels opens/transitions sessions, and leaving or moving credits closed sessions when they meet the configured minimum duration. No live server-mutating voice tests were performed.
- Landed: VC generator rules are dashboard-configured in the Community category. A source voice channel can create generated voice channels using a `{user}` name template, optional category placement, and enabled/disabled state.
- Landed: dashboard panel publishing sends a Fluxer embed to the configured panel text channel, adds reaction controls for rename, user limit, whitelist, blacklist, lock, and unlock, and records the panel in `vc_generator_control_panels`. If reaction setup fails, the panel is retained as `stale` instead of being treated as fully synced.
- Landed: joining a configured source voice channel creates a generated voice channel, records it in `generated_voice_channels`, and moves the member into it. If moving the member fails after channel creation, the generated channel is marked `orphaned`. No live server-mutating voice/channel tests were performed.
- Landed: `channel.deleted` marks matching generated voice channels deleted. Panel reactions now create durable control requests: rename, user limit, whitelist, and blacklist ask the channel owner for a follow-up reply in the panel channel, while lock and unlock apply immediately. Replies are consumed before normal bot command parsing for that user/panel channel. No live server-mutating channel/permission tests were performed.
- Landed: VC generator cleanup now tracks previous voice-channel occupancy in the Fluxer wrapper. When a voice update proves an active generated channel is empty, the bot deletes that channel and marks it `deleted`; ambiguous occupancy or platform failures leave the row active. A bot lifecycle maintenance pass also expires stale pending VC control requests. No live server-mutating channel-delete tests were performed.
- Landed: suggestion boards are dashboard-configured in the Community category. The dashboard reads bot-visible text channels, saves enabled/disabled boards, writes `bot_action_events` with `feature = suggestions`, and emits suggestions live invalidation events.
- Landed: public `suggest <idea>` posts a Fluxer embed to the default enabled suggestion board, records the `suggestions` row with board/channel/message traceability, adds up/down vote reactions, and replies to the author with the destination channel mention.
- Landed: suggestion reaction add/remove events update or remove persisted `suggestion_votes` for matching suggestion messages. Bot reactions and unrelated emoji/messages are ignored. No live server-mutating message/reaction tests were performed.
- Landed: ticket panels are dashboard-configured in the Community category. Publishing sends a Fluxer embed to the configured text channel, adds the configured open reaction, records the panel message, writes `bot_action_events` with `feature = tickets`, and emits tickets live invalidation events.
- Landed: ticket panel reactions create a durable ticket row, reserve a per-guild ticket number, add the opener as a ticket member, create a ticket text channel, apply private channel overwrites for the opener and configured staff roles, and send a short intro message. If channel creation fails, the ticket row is archived and an `open.failed` event is recorded.
- Landed: `channel.deleted` closes matching open/closed ticket rows and records a `channel.deleted` ticket event. No live server-mutating channel/permission tests were performed.
- Landed: profile builder forms are dashboard-configured in the Community category. Forms have a public URL handle, enablement, approval-required mode, and ordered text/textarea/URL fields. Config changes write `bot_action_events` with `feature = profile_builder` and emit profile-builder live invalidation events.
- Landed: `/profile-builder?guildId=...&form=...` renders enabled public forms. Submissions require a signed-in Fluxer user who belongs to the target guild, validate configured fields, store submitted values in `profile_submissions`, and write audit metadata without raw profile values.
- Landed: dashboard profile review lists recent submissions and can approve or reject pending submissions through the existing Manage Server dashboard authorization path. Review audit metadata records submission/form/user IDs and decision, not submitted field values.
- Landed: giveaways are dashboard-configured in the Community category. Publishing sends a Fluxer embed to the selected text channel, reacts with the configured entry emoji, stores the active giveaway, writes `bot_action_events` with `feature = giveaways`, and emits giveaways live invalidation events.
- Landed: giveaway reaction add/remove events track active entries for matching giveaway messages and ignore unrelated/bot reactions. Close draws persisted winners, reroll redraws from eligible entries, and cancel/close/reroll announce results back into the giveaway channel. No live server-mutating giveaway tests were performed.
- Landed: giveaway maintenance now runs from the bot lifecycle after Fluxer startup. It closes expired active giveaways in bounded batches, announces persisted winners, keeps scheduled close idempotent through a status-conditional DB claim, and retries stale bot entry-reaction setup before marking the giveaway sync active. No live server-mutating message/reaction tests were performed.
- Landed: giveaway maintenance reconciles persisted entries from the configured message reaction users through a normalized Fluxer reaction-user read wrapper. It paginates bounded reaction users, ignores bots/the NeonFlux bot user, adds missing entries, removes stale entries, and records `reactionReconciledAt` in giveaway config. No live reaction pagination/rate-limit test was performed.
- Landed: import/export now has an active Structure dashboard category. Manage Server users can export the current bot-readable role/category/channel structure, persist export snapshots, paste Fluxer-compatible structure JSON, and create persisted dry-run diffs with planned create/update/delete actions.
- Landed: reviewed structure-import dry runs can be explicitly confirmed by typing `CONFIRM <runId>`. Confirmation records approval and a dashboard audit event only.
- Landed: confirmed structure-import dry runs can run apply preflight. The preflight re-reads current server structure, detects stale baselines, separates ready creates, mapping-required creates, destructive deletes, unsupported fields, and invalid plan rows, and writes a dashboard audit event.
- Landed: confirmed, preflight-ready structure dry-runs can be applied after typing `APPLY <runId>` for supported creates, role name/permissions/color/hoist/mentionable updates, and channel/category name plus permission-overwrite updates. Apply re-runs preflight immediately before mutation, creates roles/categories/channels in dependency order, maps created-role and imported `@everyone` overwrite IDs, records source-to-created ID mappings in action details and the run apply summary, writes per-action status/details, updates the run to applied/failed, and records a dashboard audit event. Role position updates, unsupported fields, and unresolved mappings remain blocked. No live server-mutating apply test was performed.
- Landed: structure delete actions now have an explicit destructive path. Normal preflight still reports deletes as requiring destructive approval; apply accepts them only after both `APPLY <runId>` and `DELETE <runId> <deleteCount>` are typed, then re-runs preflight in destructive-approved mode and deletes channels before categories before roles. No live server-mutating delete test was performed.
- Landed: import/export structure events are now tracked by the bot for guild updates plus role/channel create/update/delete events. The handler stores only a per-guild observed-change count and latest event target in `guild_feature_settings` feature `import_export`, emits Structure page live invalidation, and shows a compact latest-change line on the Structure dashboard. It does not create automatic full snapshots for every event.
- Landed: automod rules are dashboard-configured in the Moderation category with `blocked_terms` and `invite_links` triggers plus `record`, `delete_message`, `timeout`, and `warn` actions. Rules support ignored channel/role/user IDs. Rules use dedicated `automod_rules` storage, write dashboard audit events with `feature = automod`, and emit moderation dashboard live invalidation.
- Landed: automod match events are persisted in `automod_events` from normal non-command `message.created` events. Match details store content length and match counts/terms, not raw message content or raw invite URLs. Enforcement records pending/enforced/failed/skipped status in the event details. Mutating enforcement is skipped for server owners and Manage Server authors. No live server-mutating automod tests were performed.
- Landed: automod enforcement results are now treated as handled by the message router for `recorded`, `enforced`, and `enforcement-failed` outcomes. Warn enforcement now follows the same privileged-author skip as delete/timeout before creating moderation cases.
- Landed: `channel.deleted` now fans out through VC generator cleanup, ticket cleanup, and server-event logging instead of returning before logging. DEFCON 2 bot status text now describes guarded-command restriction rather than incorrectly saying public commands are disabled.
- Landed: command-prefix changes from both the dashboard and the bot mention command now write dashboard audit events with `feature = settings`, `action = command_prefix.updated`, actor IDs, source metadata, and the saved prefix.
- Landed: dashboard live invalidation now covers server overview, invite tracking, structure import/export, and posting templates. Growth/invite/activity tables, structure import/export tables, and message templates emit Postgres dashboard notifications for the matching areas.
- Landed: posting templates are a real dashboard-only surface. Manage Server dashboard users can list saved templates, save the current plain-text/Fluxer embed payload, apply a template back into the posting form without reducing multi-embed/custom JSON payloads, and delete templates. Template save/delete writes dashboard audit events with `feature = posting`.
- Maintenance cleanup: the top-level bot feature router now delegates the `message.created` command/event pipeline to `bot-message-created-router`, and DB XP voice-session lifecycle moved to `xp-voice-sessions`. This keeps the entrypoints stable while lowering file size and responsibility overlap.
- Maintenance cleanup: the dashboard guild route test no longer drives every Community-category form submit through the full route shell. Focused server tests own those auth/audit/persistence boundaries; the route test now keeps category render/data-load coverage, reducing net added LOC by roughly 380 lines without removing feature behavior.
- Kick and unban reasons are recorded on NeonFlux cases only until the Fluxer platform exposes reason support for those operations.

## Audit Trail

- Dashboard command-access updates write `bot_action_events` with `feature = access`.
- Update action: `command_access.updated`.
- Delete action: `command_access.deleted`.
- Metadata records target type/id, user count, role count, source, and actor display metadata when Fluxer identity lookup succeeds.

## Test Volume Notes

- `apps/bot/src/bot-feature-router.test.ts` is the largest test file because it verifies dispatch order across commands, events, mode gates, DEFCON, and feature handler priority. Do not add feature-internal behavior there unless routing order is the behavior under test.
- `apps/web/src/routes/dashboard/-guild.test.ts` is the largest dashboard test file because it still carries detailed Posting/embed-builder behavior. Trim this only after moving builder behavior to focused component tests; otherwise color picker, empty-field omission, content-only, embed-only, advanced JSON, and preview coverage would be lost.
- New feature tests should stay lean: DB contract, server auth/audit boundary, and bot handler safety behavior. Avoid route-level duplicates when focused tests already prove the feature internals.
